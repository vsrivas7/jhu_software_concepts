# Module 5 — GradCafe Flask App (Hardened)

A Flask + PostgreSQL web app that scrapes and analyzes GradCafe admission data,
hardened against SQL injection and following software assurance best practices.

## Fresh Install

### Option A — pip + venv (required)

```bash
# 1. Clone / enter the project
cd module_5

# 2. Create and activate virtual environment
python -m venv venv
source venv/bin/activate        # Windows: venv\Scripts\activate

# 3. Install all dependencies
pip install -r requirements.txt

# 4. Install the project in editable mode
pip install -e .

# 5. Copy env template and fill in your credentials
cp .env.example .env
# edit .env with your real DB values

# 6. Run the app
flask --app src.flask_app:create_app run
```

### Option B — uv (required alternate)

```bash
# 1. Install uv if you haven't already
pip install uv

# 2. Create a venv and sync exactly to requirements.txt
cd module_5
uv venv
source .venv/bin/activate       # Windows: .venv\Scripts\activate
uv pip sync requirements.txt

# 3. Install project in editable mode
uv pip install -e .

# 4. Copy env template
cp .env.example .env
# edit .env with your real DB values

# 5. Run the app
flask --app src.flask_app:create_app run
```

> `uv pip sync` makes the environment match `requirements.txt` exactly —
> it removes packages not listed there, giving stronger reproducibility than `pip install`.

## Generating the Dependency Graph

Requires [Graphviz](https://graphviz.org/download/) (`dot` command available on PATH).

```bash
pip install pydeps
pydeps src/flask_app.py --noshow -T svg -o dependency.svg
```

The output `dependency.svg` is saved in `module_5/`.

### Dependency Graph Summary

The dependency graph for `flask_app.py` reveals five key relationships.
`flask_app` sits at the center and imports from three sibling modules: `scrape`,
`load_data`, and `query_data`, which represent the three core concerns of the
application (data collection, persistence, and analysis). `scrape` depends on
`requests` and `beautifulsoup4` to fetch and parse HTML from The Grad Cafe website.
`load_data` and `query_data` both depend on `psycopg` and its `sql` submodule for
safe, parameterised database access. `flask_app` itself depends on `Flask` for
routing and `os` for reading environment variables, keeping secrets out of source
code. Together the graph shows a clean, layered architecture with no circular
dependencies.

## Environment Variables & DB Hardening

Copy `.env.example` to `.env` and fill in your values. **Never commit `.env`.**

```
DATABASE_URL=postgresql://gradcafe_user:yourpassword@localhost:5432/gradcafe
DB_HOST=localhost
DB_PORT=5432
DB_NAME=gradcafe
DB_USER=gradcafe_user
DB_PASSWORD=yourpassword
```

### Least-Privilege DB User Setup

The application only reads from and inserts into the `applicants` table.
Run the following SQL as a superuser to create a least-privilege role:

```sql
-- Create a restricted user (not a superuser)
CREATE USER gradcafe_user WITH PASSWORD 'yourpassword';

-- Grant connection access to the database
GRANT CONNECT ON DATABASE gradcafe TO gradcafe_user;

-- Grant usage on the schema
GRANT USAGE ON SCHEMA public TO gradcafe_user;

-- Grant only SELECT and INSERT on the specific table
-- (no UPDATE, DELETE, DROP, ALTER, or owner-level permissions)
GRANT SELECT, INSERT ON TABLE applicants TO gradcafe_user;
```

**Why these permissions only?**
The scraper inserts new rows (`INSERT`) and the analysis page reads them (`SELECT`).
No feature of the app requires `UPDATE`, `DELETE`, `DROP`, or `ALTER`, so those
are intentionally withheld. If the user account is compromised, an attacker cannot
drop tables, modify schema, or exfiltrate other databases.

## SQL Injection Defenses

All SQL in `query_data.py` and `load_data.py` uses **psycopg SQL composition**:

| Technique | Used for | Example |
|-----------|----------|---------|
| `sql.SQL(...)` | Statement structure | `sql.SQL("SELECT ... FROM {table} WHERE ... LIMIT %s")` |
| `sql.Identifier(...)` | Table / column names | `.format(table=sql.Identifier("applicants"))` |
| `%s` + params tuple | All user-supplied values | `cur.execute(stmt, ("%fall%", safe_limit))` |

**What changed from module_4 and why it's safe:**

- Removed all f-strings and string concatenation from SQL construction.
- Dynamic identifiers (table/column names) now go through `sql.Identifier`, which
  psycopg quotes safely — preventing identifier injection attacks.
- User-supplied values **never** appear in SQL text; they are passed as a separate
  `params` argument to `cursor.execute()`, so the database driver handles escaping.
- Every query has an explicit `LIMIT %s` with a `_clamp_limit()` helper that clamps
  the value between 1 and 100, preventing unbounded data dumps.
- SQL statement construction (the `sql.SQL(...)` object) is fully separated from
  execution (`cur.execute(stmt, params)`), making the distinction clear and auditable.

## GitHub Actions CI

The workflow at `.github/workflows/ci.yml` runs on every push and pull request
with four separate jobs:

1. **Pylint** — fails if score is below 10.00/10
2. **Pytest** — runs full test suite with coverage
3. **Dependency graph** — generates `dependency.svg` and fails if missing
4. **Snyk** — runs `snyk test` against `requirements.txt`
